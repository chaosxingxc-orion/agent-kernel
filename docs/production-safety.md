# Production Safety

本文档定义 agent-kernel 在生产环境中的安全使用边界，重点是避免把 mock / PoC / test-only 实现误装配到线上。

## 1. 生产禁用清单

以下实现或运行方式不应作为生产默认配置，也不应在生产环境中作为 fallback 自动启用：

- `echo` 类实现
  - 典型特征：只回显输入、不接入真实后端能力。
  - 风险：会让系统看起来“可用”，但实际上不会产生真实效果。
- `in_process_python`
  - 典型特征：在进程内直接执行 Python 片段或脚本。
  - 风险：安全边界弱，容易引入任意代码执行、隔离不足和可观测性缺失问题。
- `InMemory*`
  - 典型特征：状态只存在于进程内，重启即丢失。
  - 风险：不具备生产所需的持久性、恢复能力和横向扩展稳定性。
- `Static*`
  - 典型特征：静态桩、静态策略、固定返回值或 PoC 级实现。
  - 风险：无法反映真实业务状态，容易掩盖故障、回放和一致性问题。

建议把上述实现视为以下用途：

- 单元测试
- 本地开发
- PoC 验证
- 临时调试

不建议用于生产流量、生产任务编排或长期运行的后台服务。

## 2. 推荐生产后端

生产环境优先选择**持久化后端**，默认推荐：

- `sqlite`：适合单机、边缘部署、轻量化生产场景，配置简单，具备持久化能力。
- 其他持久化存储：只要满足持久化、恢复、并发控制和可观测性要求，也可以作为生产后端。

推荐原则：

- 生产环境优先选“可恢复、可审计、可重放”的后端。
- 默认装配链路应避免落到内存态或静态桩。
- 如果后端选择依赖环境变量或配置文件，必须显式声明生产值，不能依赖隐式默认值。

## 3. 配置建议

### 3.1 启动即校验

生产启动时应校验后端类型，若检测到 `echo`、`in_process_python`、`InMemory*` 或 `Static*`，应直接失败退出，而不是继续运行。

### 3.2 显式配置

建议所有生产配置都采用显式声明，至少包括：

- 后端类型
- 持久化路径或连接信息
- 超时与重试策略
- 诊断/审计日志位置

### 3.3 禁止自动回退

不要把生产默认值设成“先用 mock，再自动切换真实后端”。这种策略会让故障在上线后被静默掩盖，排障成本很高。

### 3.4 变更控制

任何会改变后端类型的改动，都应当经过：

- 配置评审
- 环境校验
- 启动验证
- 回滚预案确认

## 4. 风险说明

### 4.1 功能风险

`echo`、`Static*` 和 `InMemory*` 往往能让接口“看起来通了”，但不会提供生产所需的真实状态管理和故障恢复能力。

### 4.2 数据风险

`InMemory*` 在进程重启、滚动发布、实例漂移时会丢失状态，可能造成任务重复、状态回退或审计缺口。

### 4.3 安全风险

`in_process_python` 会把执行面暴露在进程内部，扩大攻击面，也更难做权限隔离和行为审计。

### 4.4 运维风险

静态桩和内存态实现会让健康检查、告警和容量评估失真，导致生产问题在观测层面被延迟发现。
